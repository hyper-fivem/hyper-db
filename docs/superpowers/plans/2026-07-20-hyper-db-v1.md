# hyper-db v1 Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the hyper-db PRD v1 foundation: a bun-workspaces monorepo with a type-safe schema DSL + Query AST (pg-core & mysql-core), a TS/Node-22 core engine (query pipeline, declarative cache, hot-store with write-behind, locks/rate-limit, pub/sub, error model, stats), a codegen CLI (TS/Lua/C# + migrations), Lua & C# client runtimes, and a FiveM resource bundle — all tested with `bun test`.

**Architecture:** Single-owner core resource holds all PG/MariaDB/Redis connections; consumers cross the runtime boundary only with `queryId + flat params[]`. Schema is the single source of truth (TS); codegen projects typed APIs into Lua/C#/TS. Drivers are abstracted behind interfaces so the entire engine is unit-testable with in-memory fakes; real adapters use postgres.js / mariadb / ioredis.

**Tech Stack:** TypeScript (Node 22 target), bun workspaces + bun test, postgres.js, ioredis, mariadb, esbuild (via `bun build`), docker-compose.

## Global Constraints

- Always `bun` / `bunx`, never npm/npx (user rule).
- License: GPL-3.0 (repo LICENSE already present).
- Runtime boundary payload: `queryId: string` + flat `params: unknown[]` only — no nested payloads.
- Dialects are separate typed modules (`pg-core`, `mysql-core`) — no common-denominator tax.
- Cache is fully opt-in/declarative; TTL is always an upper bound.
- Node 22 core (`node_version '22'` in fxmanifest), esbuild bundle (no FiveM webpack).
- All engine logic unit-testable without live DBs (driver interfaces + fakes); real integration via docker-compose.
- Errors cross the boundary as `{ code, message, details }` — typed exceptions per language.
